## Аннотация

Документ представляет применение закона архитектурной согласованности к организации программного кода. На примере рефакторинга REST API показано, как принципы устойчивости сложных систем ($E_{otkl} < c_0$, $K_{ij} > |\omega_i - \omega_j|$, $\Phi \in [0.8, 1.2]$) переводятся в конкретные правила проектирования кода. Предлагаются критерии оценки архитектурной согласованности программных модулей и методы её достижения через разделение ответственности, выделение сервисного слоя и переиспользование функций.

**Ключевые слова:** рефакторинг, архитектурная устойчивость, разделение ответственности, программная инженерия, закон архитектурной согласованности

## 1. Закон архитектурной согласованности для кода

### 1.1. Формулировка

Программный модуль остаётся поддерживаемым и расширяемым лишь до тех пор, пока отклонение от принципов чистой архитектуры не превышает критический порог:

$$E_{otkl}^{code} < c_0^{code}$$

где:
- $E_{otkl}^{code}$ — мера отклонения кода от принципов: единственной ответственности, открытости/закрытости, разделения слоёв
- $c_0^{code}$ — критический порог, после которого код становится «сложным для понимания» и «страшным для изменения»

### 1.2. Признаки нарушения

| Признак | Описание | Последствие |
|---------|----------|-------------|
| Смешение слоёв | Маршрутизация, бизнес-логика, работа с БД в одном файле | Невозможно тестировать, трудно менять |
| Дублирование | Один и тот же код в нескольких местах | Ошибка исправляется в одном месте, но не в других |
| Высокая связанность | Изменение в одном месте требует изменений во многих | Риск поломки при каждом доработке |
| Низкая связность | Одна функция делает много разных вещей | Непонятно, что она делает |

## 2. Критерии архитектурной согласованности кода

### 2.1. Разделение ответственности (аналог $\Phi$)

$$\Phi_{code} = \frac{L_{layer} \cdot R_{reuse}}{C_{coupling}}$$

| Символ | Что это | Хорошее значение |
|--------|---------|------------------|
| $L_{layer}$ | Количество чётко выделенных слоёв (маршруты → сервисы → репозитории) | $\ge 3$ |
| $R_{reuse}$ | Доля переиспользуемых функций от общего объёма | $\ge 0.5$ |
| $C_{coupling}$ | Связанность (количество зависимостей между модулями) | $\le 5$ на модуль |

**Диапазон устойчивости:** $\Phi_{code} \in [0.8, 1.2]$

### 2.2. Контроль синхронизации (аналог $K_{ij} > |\omega_i - \omega_j|$)

$$K_{ij}^{code} = \frac{\text{вызовы из i в j} + \text{вызовы из j в i}}{\text{общее число вызовов}}$$

**Условие устойчивости:** $K_{ij}^{code} \le 0.3$ для любых двух модулей

**Что это значит:** если два модуля вызывают друг друга слишком часто (>30% всех вызовов), они «синхронизируются». Изменение одного будет ломать другой.

### 2.3. Показатель Херста для кода ($H_{code}$)

$H_{code}$ — мера того, накапливаются ли технические долги:

| $H_{code}$ | Состояние |
|------------|-----------|
| $< 0.6$ | Код чистый, техдолг не накапливается |
| $0.6 - 0.7$ | Появляются «костыли», требует рефакторинга |
| $\ge 0.7$ | Критический техдолг, риск поломки при любом изменении |

## 3. Пример рефакторинга: до и после

### 3.1. До рефакторинга (наружение)

```python
# routes/orders.py — весь код в одном файле

from flask import Flask, request
from database import db

app = Flask(__name__)

@app.route('/orders', methods=['POST'])
def create_order():
    data = request.get_json()
    
    # Валидация (смешана с логикой)
    if not data.get('user_id') or not data.get('items'):
        return {"error": "Invalid input"}, 400
    
    # Проверка пользователя (прямой запрос в БД)
    user = db.users.find_one({"_id": data['user_id']})
    if not user:
        return {"error": "User not found"}, 404
    
    # Расчёт суммы (бизнес-логика)
    total = 0
    for item in data['items']:
        product = db.products.find_one({"_id": item['product_id']})
        if product:
            total += product['price'] * item['quantity']
    
    # Сохранение заказа (прямой запрос в БД)
    order = {
        "user_id": user['_id'],
        "total": total,
        "items": data['items'],
        "status": "created"
    }
    result = db.orders.insert_one(order)
    
    return {"order_id": str(result.inserted_id), "total": total}
```

### 3.2. После рефакторинга (согласованность)

**routes/orders.py — только маршрутизация**

```python
from flask import Flask, request
from services import order_service

app = Flask(__name__)

@app.route('/orders', methods=['POST'])
def create_order():
    """Маршрут теперь очень короткий — только принять запрос и вернуть ответ."""
    data = request.get_json()
    result = order_service.create_order(data)
    return result
```

**services/order_service.py — бизнес-логика**

```python
def create_order(input_data):
    """Сервис содержит логику, но она хорошо структурирована по шагам."""
    # 1. Валидация
    if not validate_order_input(input_data):
        return {"error": "Invalid input"}, 400
    
    # 2. Проверка прав (можно вынести в отдельный сервис auth_service)
    user = get_user_or_fail(input_data['user_id'])
    
    # 3. Бизнес-логика (вызов другой функции)
    total = calculate_order_total(input_data['items'])
    
    # 4. Работа с БД (вызов функции-репозитория)
    order_id = save_order_to_db(user.id, total, input_data['items'])
    
    return {"order_id": order_id, "total": total}

def validate_order_input(data):
    """Теперь эту функцию можно переиспользовать в других местах."""
    return data.get('user_id') and data.get('items')

def calculate_order_total(items):
    """Расчёт суммы — отдельная переиспользуемая функция."""
    total = 0
    for item in items:
        product = get_product_by_id(item['product_id'])
        if product:
            total += product['price'] * item['quantity']
    return total

def save_order_to_db(user_id, total, items):
    """Сохранение заказа — отдельная функция."""
    order = {
        "user_id": user_id,
        "total": total,
        "items": items,
        "status": "created"
    }
    return db.orders.insert_one(order).inserted_id
```

**services/product_service.py — работа с продуктами (выделено)**

```python
def get_product_by_id(product_id):
    """Функция, которую можно вызывать из любого места."""
    return db.products.find_one({"_id": product_id})
```

**services/user_service.py — работа с пользователями (выделено)**

```python
def get_user_or_fail(user_id):
    user = db.users.find_one({"_id": user_id})
    if not user:
        raise ValueError("User not found")
    return user
```

## 4. Принципы архитектурной согласованности кода

### 4.1. Разделяй по ответственности

| Слой | Что содержит | Что НЕ содержит |
|------|--------------|-----------------|
| Маршруты (routes) | Только приём запроса, вызов сервиса, возврат ответа | Бизнес-логику, работу с БД |
| Сервисы (services) | Бизнес-логику, вызов репозиториев | Прямые запросы к БД, HTTP-детали |
| Репозитории (repositories) | Только работу с БД | Бизнес-логику |
| Утилиты (utils) | Переиспользуемые функции | Зависимости от контекста |

### 4.2. Не допускай синхронизации

- Два модуля не должны вызывать друг друга часто
- Если $K_{ij}^{code} > 0.3$ — выноси общую логику в третий модуль

### 4.3. Контролируй техдолг через $H_{code}$

- Регулярно пересчитывай $H_{code}$ по истории коммитов
- Если $H_{code}$ растёт — планируй рефакторинг

## 6. Связь с математическим ядром

| Признак | Влияние на математические параметры |
|---------|-----------------------------------|
| Смешение слоёв | $DD \uparrow$, $EVS \downarrow$ |
| Высокая связанность | $K_{ij} \uparrow$, приближение к $|\omega_i - \omega_j|$ |
| Низкая связность | $H \downarrow$ (энтропия альтернатив) |
| Дублирование | $r_{rollback} \uparrow$ |

## 7. Заключение

Закон архитектурной согласованности применим не только к физике плазмы и катализу, но и к программному коду.

Критерии:
- $\Phi_{code} \in [0.8, 1.2]$ — разделение слоёв и переиспользование
- $K_{ij}^{code} \le 0.3$ — контроль связей между модулями
- $H_{code} < 0.6$ — предотвращение накопления техдолга
- $E_{otkl}^{code} < c_0^{code} = 0.75$ — общее отклонение ниже порога

Рефакторинг — это не «косметика», а восстановление архитектурной устойчивости системы.
