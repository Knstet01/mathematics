## 1. Целевая архитектура

### 1.1. Общий принцип

Не строить «доступ к ChatGPT». Строить **ваш продукт**, который:
- Вызывает OpenAI API
- Применяет поверх ответа **«рамку» как интерпретационный слой**
- Хранит историю, управляет контекстом, контролирует лимиты и безопасность

**Почему:** иначе вы упрётесь в условия использования OpenAI и в отсутствие контроля над контекстом и данными.

## 2. Интерпретация через рамку (Framework Interpreter)

### 2.1. Двухпроходная архитектура

**Pass A — «Raw Answer»**
- Модель отвечает на запрос пользователя максимально прямо
- Фиксируются: `raw_text`, источники/инструменты, метрики

**Pass B — «Framework Rendering»**
- Отдельный вызов модели (или локальный рендерер правилами)
- **Не меняет смысл** ответа
- Помечает допущения, переводит ответ в формат рамки:
  - Определения
  - Оси (категории анализа)
  - Инварианты
  - Условия применимости
  - Риски ложной универсальности

**Важно:** хранить обе версии. Показывать пользователю `final` (Pass B), но иметь `raw` для отладки/проверки.

### 2.2. Техническая реализация Pass B

**MVP:** второй вызов модели с жёстким контрактом через JSON Schema.

```json
{
  "name": "framework_rendering",
  "strict": true,
  "schema": {
    "type": "object",
    "properties": {
      "raw_summary": {"type": "string"},
      "framework_axes": {
        "type": "array",
        "items": {
          "type": "object",
          "properties": {
            "axis": {"type": "string"},
            "value": {"type": "string"},
            "confidence": {"type": "number"}
          }
        }
      },
      "assumptions": {"type": "array", "items": {"type": "string"}},
      "applicability_limits": {"type": "array", "items": {"type": "string"}},
      "risks": {"type": "array", "items": {"type": "string"}}
    }
  }
}
```

**Перспектива:**
- Выделенный fine-tuned слой для снижения затрат
- Локальный рендерер на основе правил (без вызова модели)

### 2.3. Ограничения Pass B

- Pass B **не добавляет факты**
- Если Pass B противоречит raw answer — приоритет у raw
- Расхождения логируются для доработки рамки

## 3. Контекст, память, безопасность

### 3.1. Память (ваша, не «магия модели»)

Храните:
- **Краткая state summary** (1–3 KB) на сессию
- **Долговременные факты** — только по явному согласию пользователя
- **Версии рамки** и её параметры

### 3.2. Конфиденциальность

- По умолчанию: **не логировать содержимое**, только метрики
- Отдельный режим `diagnostics opt-in` для сохранения диалогов (по согласию)

### 3.3. Политики и модерация

**Двухуровневая модерация:**
- **На входе:** отсев запросов, нарушающих OpenAI Usage Policies
- **На выходе:** проверка ответа перед показом пользователю

**Использовать:**
- OpenAI Moderation API
- Дополнительные правила на уровне вашего Policy & Moderation сервиса

**Ссылки:**
- [OpenAI Usage Policies](https://openai.com/policies/usage-policies)
- [OpenAI Business Terms](https://openai.com/policies/business-terms) — для коммерческого API-использования

## 4. Биллинг и лимиты

### 4.1. Модель расходов

```
total_cost = tokens_in_A × cost_A + tokens_out_A × cost_A
            + tokens_in_B × cost_B + tokens_out_B × cost_B
            + overhead (модерация, стриминг)
```

### 4.2. Оптимизация

- Pass B запускать только когда нужен «рамочный» вывод (вероятно, всегда — тогда оптимизировать промпт и длину)
- Резать контекст агрессивно: `summary` вместо полной переписки
- Кэшировать частые запросы

### 4.3. Планы (предварительно)

| План     | Особенности                                            |
|------    |-------------                                           |
| **Free** | Короткие ответы, без истории, ограничение токенов      |
| **Pro**  | Длинные ответы, история диалогов, приоритетная очередь |
| **Team** | Shared workspace, аудит, единый биллинг                |

## 5. Критические требования к качеству

| Требование | Реализация |
|------------|------------|
| **Идемпотентность запросов** | Уникальный `idempotency_key` на каждый запрос; повторные вызовы не создают новые расходы |
| **Версионирование промптов/рамки** | Semver (v1.0.0). Каждая версия определяет правила интерпретации, пороги, допустимые переходы |
| **Детерминизм формата** | JSON Schema для Pass B, фиксированные секции для Pass A |
| **Наблюдаемость** | Метрики: latency (p50/p95/p99), error rate, cost per session, drop-offs |
| **План миграций API** | Responses API — основной путь. Следить за депрекациями (Assistants API выключается 26 Aug 2026) |

## 6. Мониторинг расхождений и обратная связь

Система должна логировать случаи, когда Pass B:
- Существенно меняет смысл ответа
- Вводит неверные интерпретации
- Противоречит raw answer

Это данные для:
- Доработки рамки
- Обучения интерпретационного слоя
- Аудита качества

## 7. MVP-план

### MVP-0 (1–2 недели разработки ядра)

- Web-only
- Login + простая подписка (Stripe/Paddle)
- Responses API + стриминг (SSE)
- Pass A / Pass B (второй вызов модели с JSON Schema)
- Хранилище сессий (PostgreSQL)
- Базовая модерация

### MVP-1 (мобильные клиенты)

- iOS/Android тонкие клиенты к вашему backend
- Push-уведомления
- Офлайн-кэш последних N диалогов

### Пост-MVP

- Выделенный fine-tuned слой для Pass B
- Локальный рендерер правилами
- Режим проверки (показывать raw ответ рядом с framework-rendered)
- Команды и shared workspace

---

## 8. Версионирование рамки

Рамка имеет версию (semver). Каждая версия определяет:
- Набор правил интерпретации
- Пороговые значения
- Допустимые переходы

**Принцип:** обновление интерпретационного слоя не ломает старые сессии. Каждая сессия привязана к версии рамки, актуальной на момент создания.

---

## 9. Схема потока данных

```
1. Client → API Gateway (auth, rate limit, idempotency_key)
2. Gateway → Orchestrator
3. Orchestrator → Conversation Store (загрузка контекста, версии рамки)
4. Orchestrator → Policy & Moderation (входная проверка)
5. Orchestrator → OpenAI Responses API (Pass A)
6. Orchestrator → Framework Interpreter (Pass B, если нужен)
7. Orchestrator → Policy & Moderation (выходная проверка)
8. Orchestrator → Conversation Store (сохранение сессии, raw + final)
9. Orchestrator → Analytics (метрики)
10. Orchestrator → Client (стриминг или полный ответ)
```

## 10. Связь с математическим ядром архитектурной устойчивости

Framework Interpreter опирается на формальное ядро:

| Элемент рамки | Применение в интерпретационном слое |
|---------------|-------------------------------------|
| $E_{otkl} < c_0$ | Оценка отклонения ответа от допустимого диапазона |
| $K_{ij}$ | Контроль связей между элементами интерпретации |
| $H$ | Мониторинг накопления необратимости в диалоге (рост жёсткости ответов) |
| $DD$ | Глубина зависимостей в рассуждении (многоуровневая интерпретация) |
| $MTTR\text{-}E$ | Эластичность восстановления при ошибках (как быстро система возвращается к корректной работе) |
